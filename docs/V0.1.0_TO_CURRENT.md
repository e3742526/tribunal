# v0.1.0 to current development state

This document compares the first published Tagteam release, `v0.1.0`, with
the current `main` branch.

| Baseline | Value |
|---|---|
| Published release | `v0.1.0` |
| Release commit | `829be86c3e79b4b5676c65b738e4bb2a1b4ff08e` |
| Release date | 2026-07-08 |
| Current commit | `77a14b523779e83cfdf26f7bd80a695199e12245` |
| Current date | 2026-07-11 |
| Commits since release | 51 |
| Tree delta | 111 files, 24,187 insertions, 8,676 deletions |
| Go test functions | 198 at release; 357 currently |
| Go source files | 26 at release; 100 currently |

The current tree still describes itself as the `v0.1.4` development surface
in user documentation, but there is no corresponding published Git tag in the
repository at the time of writing. This is therefore a release-to-development
comparison, not release notes for another published version.

## Executive summary

`v0.1.0` already contained the four orchestration modes, adapter abstraction,
review loops, package slicing, layered configuration, persisted run artifacts,
and the initial safety model. Current `main` is not a new orchestration concept;
it is a substantial operational hardening and usability expansion of that
foundation.

The largest changes are:

- a launch-capable, role-aware TUI instead of CLI-only operation;
- authoritative run state outside the agent worktree;
- phase-aware resume and recovery for interrupted or partially edited runs;
- persistent findings across rounds and historical runs;
- host-enforced worker/reviewer contracts, integrity checks, and transfer gates;
- live progress, invocation delivery records, watchdogs, budgets, and fallback
  telemetry;
- hardened Claude JSON recovery and cross-process invocation serialization;
- verified binary/manifest enforcement for commands that mutate repositories;
- newer, mode-specific default model assignments and clearer role-selection
  flags.

## User-visible command surface

### v0.1.0

The release exposed the positional run command plus:

```text
review
fix
status
plan
transcript
doctor
init
```

It supported supervisor, relay, solo, and adversarial modes, but did not expose
an explicit `run` alias, interactive TUI, persisted-run continuation, findings
inspection, gated patch transfer, or installation verification commands.

### Current

The current command surface adds:

| Command | Difference from v0.1.0 |
|---|---|
| `run` | Explicit alias for the positional prompt path, matching conventional agent CLI usage. |
| `resume [RUN_ID]` | Verifies and continues an interrupted run from persisted phase state. |
| `findings [--all]` | Lists unresolved findings across every persisted run for the repository; `--all` includes disposition history. |
| `findings defer` | Records an operator-approved blocker/major deferral with a required reason. |
| `transfer RUN_ID` | Revalidates and applies a run patch to a primary checkout without staging or committing it. |
| `tui [RUN_ID]` | Launches and monitors runs in an interactive terminal dashboard. |
| `verify-install` | Verifies embedded build identity and the adjacent checksum manifest. |
| `version` | Prints structured build identity rather than relying only on Cobra's version flag. |

Shell completion remains provided by Cobra.

## TUI and interaction model

`v0.1.0` was effectively CLI-only. Current `main` includes a full-screen TUI
with four primary surfaces:

1. A sparse compose screen for entering the task.
2. A Team panel for selecting orchestration mode and assigning a model to each
   active role.
3. A Settings panel for execution policy, scope, rounds, timeouts, tests, lint,
   slicing, retrieval, dirty-worktree behavior, and JSON repair.
4. A scrollable run-detail view for phase, roles, plan, findings, diff status,
   tests, and artifact paths.

The TUI uses role-first model selection. In a supervisor run, for example, it
labels the implementation target as `worker` and the planning/review target as
`supervisor`; relay and adversarial modes expose their own role names rather
than presenting one ambiguous global model selector.

The slash-command palette is contextual and includes model, profile, mode,
provider effort, scope, timeout, watchdog, test, lint, retrieval, and run-watch
controls. Suggestions appear on demand instead of occupying the idle screen.

Active runs now open directly in detail view. The TUI polls the shared snapshot
surface once per second and shows logical role, phase, elapsed/idle time, live
working-diff totals, and queued Claude lock waits. Completed history remains
compose-first unless the operator explicitly opens a run.

## Orchestration behavior

### Modes retained

Both versions support:

- `supervisor`: supervisor plans/reviews; worker edits;
- `relay`: scout supplies advisory context; coder edits; supervisor reviews;
- `solo`: one editor with no claim of independent review;
- `adversarial`: coder and adversary iterate over implementation and review.

### Current hardening

Current orchestration adds explicit host-owned state and policy around those
modes:

- Role targets are resolved by mode instead of treating coder/adversary names
  as universal slots.
- Runtime role replacement is recorded in `role_losses` and `role_statuses`,
  including selected fallbacks and failure reason codes.
- A worker failure before useful edits can engage configured replacement;
  partial edits require recovery arbitration or quarantine instead of blindly
  replaying another writer.
- Relay scout output remains advisory. Host state records scout execution,
  context-budget decisions, retrieval evidence, and degraded continuation.
- Supervisor work plans are persisted and package progress survives resume.
- Round exhaustion collects final no-edit reports rather than continuing
  indefinitely or silently declaring success.
- Optional invocation and wall-clock budgets are persisted in final output;
  role invocations are counted even when the configured limit is unlimited.
- The default no-progress watchdog cancels an invocation after five minutes
  without Git or output activity, subject to the shorter effective invocation
  timeout.

## Default roles and models

Defaults changed materially and are now centralized as mode-specific presets.

| Role/mode | v0.1.0 | Current |
|---|---|---|
| Supervisor-mode worker | `agy:Gemini 3.5 Flash (High)` | `agy:Gemini 3.5 Flash (Medium)` |
| Supervisor-mode supervisor | `claude:opus` | `codex:gpt-5.6-sol`, high reasoning effort |
| Relay scout | `agy:gemini-3.5-flash-low` | `openai-compatible:gemma4:latest` |
| Relay coder | `codex:gpt-5.4-mini` in the built-in relay profile | `agy:Gemini 3.5 Flash (Medium)` |
| Relay supervisor | `claude:sonnet` in the built-in relay profile | `codex:gpt-5.6-sol` |
| Adversarial coder | bare `codex` or profile-specific `gpt-5-codex-mini` | `codex:gpt-5.6-terra` |
| Adversarial reviewer | bare `claude` / `claude:opus` | `claude:claude-opus-4-8` |
| Claude adapter default | `sonnet` | `claude-sonnet-5`, high effort |
| Agy adapter default | `gemini-3.5-flash` | `Gemini 3.5 Flash (Medium)` |
| Local OpenAI-compatible default | unset | `http://127.0.0.1:11434/v1`, `gemma4:latest` |

The current default supervisor flow is Claude-free. It also includes a default
worker fallback to `codex:gpt-5.6-terra` and a `claude-failover` profile that
maps common Claude targets to Codex replacements.

Relay retrieval changed from enabled by default in `v0.1.0` to disabled by
default. It remains available as bounded, host-owned local retrieval.

Model names reflect the configured development environment and can drift as
vendor CLIs evolve. They are not a compatibility promise from Tagteam.

## Configuration and role selection

The precedence model remains:

```text
flags > shell TAGTEAM_* env > workdir .env > repo .tagteam.toml > user config > built-ins
```

Current `main` expands the configuration surface with:

- mode-aware `--model`, `--worker`, `--supervisor`, `--reviewer`, `--coder`,
  and `--scout` flags while retaining `-mc`/`-ma` compatibility;
- separate Codex reasoning effort and Claude effort settings;
- `state_root`, watchdog timeout, lint command, churn thresholds, JSON repair,
  fallback ladders, target-specific fallbacks, and loss policies;
- explicit `relay_coder` so relay and adversarial implementation defaults do
  not overload the same setting;
- profile controls in both CLI and TUI;
- `adapters.claude.serialize` / `TAGTEAM_CLAUDE_SERIALIZE`;
- context and reserved-output budgets for adapters;
- stricter removal of high-authority settings from untrusted repo config.

Repo-local configuration remains partially trusted by default. Tests, command
passthrough, permission/tool widening, state-root changes, fallback policy, and
adapter endpoint credentials require an appropriate trusted source.

## State storage and lifecycle

### v0.1.0

Run artifacts were rooted under the checkout's `.tagteam/runs/<run-id>` tree.
That made state easy to locate but exposed host-owned orchestration files to
editing agents and could dirty or conflict with the worktree.

### Current

Authoritative state defaults to:

```text
~/.local/state/tagteam/<repo-id>/runs/<run-id>/
```

The checkout retains only a small `.tagteam/repo.json` pointer. A centralized
`StateLocator` resolves repository identity, state root, active/latest pointers,
run directories, and legacy migration behavior.

Lifecycle additions include:

- external `active.json` for in-flight discovery;
- `latest.json` only after authoritative finalization;
- phase-aware `state.json` and append-only `events.jsonl`;
- atomic/durable JSON and stream writes;
- run locks and invocation records;
- explicit passed, degraded, blocked, failed, cancelled, and quarantined
  terminal states;
- recovery of stale active pointers and interrupted phases;
- compatibility reads for legacy in-worktree state where safe.

This is the most important storage compatibility change. Tools that assumed
`.tagteam/runs` is authoritative should switch to CLI status/snapshot commands
or resolve the repository pointer.

## Resume, recovery, and partial edits

`v0.1.0` did not expose a general resume command. Current `resume` validates
the saved baseline, repository identity, diff hash, protected artifacts, and
phase prerequisites before continuing the first incomplete phase.

Current recovery behavior distinguishes:

- failures before an editor changed the worktree;
- failures after partial edits;
- reviewer/output-contract failures;
- interrupted host test or review phases;
- integrity violations requiring quarantine.

Partial edits are preserved as evidence. A recovery decision can accept,
replace, or block the next writer; Tagteam does not silently stack another
agent on top of ambiguous partial state.

## Findings and review contracts

`v0.1.0` looped review findings back into the editor, but current `main` makes
the finding lifecycle durable:

- `findings.json` persists stable finding IDs and statuses across rounds;
- omitted blocker/major findings remain open until explicitly disposed;
- dispositions require evidence (`fixed`, `disputed_with_evidence`, or an
  operator-approved deferral);
- `tagteam findings` aggregates unresolved findings across historical runs,
  preventing a clean latest run from hiding older open findings;
- transfer is blocked while unresolved blocker/major findings remain.

Review schema v2 adds explicit data-loss checks for malformed-input
preservation, annotation/history retention, ambiguous identity handling, and
reviewer read-only non-mutation. A `pass` verdict cannot coexist with blocking
findings or failed required checks.

Worker output is also contractual. Implementations must report status, summary,
changed files, checks run, and remaining risks. Host validation compares the
declared files with the actual worktree and rejects misleading completion.

## Quality, integrity, and transfer gates

Current `main` adds host-enforced gates that were not complete in `v0.1.0`:

- protected run-artifact integrity snapshots before and after invocations;
- read-only reviewer mutation detection;
- deterministic tracked, staged, untracked, renamed, deleted, and binary diff
  capture through a temporary Git index;
- diff hashes and persisted file/numstat metadata;
- allowed-path scope enforcement;
- churn limits for files, changed lines, fixtures, whitespace ratio, and
  minimum semantic density;
- bounded command output and durable stdout/stderr delivery streams;
- test identity checks and preserved test evidence;
- transfer-time repository identity, baseline, cleanliness, patch-hash, lint,
  test, findings, regression, and final-review verification.

Transfer is never automatic. It applies a validated patch without staging or
committing it, leaving the operator in control of the destination checkout.

## Live observability

`v0.1.0` primarily exposed completed artifacts and basic status. Current
observability includes:

- active-run discovery before `final.json` exists;
- one-second TUI/status snapshots;
- role, phase, current round, elapsed time, and no-progress time;
- tracked plus untracked working-diff file/addition/deletion totals;
- invocation stdout/stderr byte counts and delivery records;
- timeout calibration evidence;
- queued Claude lock status, queue elapsed time, adapter, and holder PID;
- explicit degraded and blocking reasons in text and JSON output.

The same `RunSnapshot` assembly backs `tagteam status` and the TUI, reducing
disagreement between interactive and scripted monitoring.

## Claude behavior

`v0.1.0` documented vendor variability but did not absorb several observed
Claude failure modes. Current behavior adds:

- clear surfacing of Claude envelopes with `is_error` or `error_*` subtypes;
- bounded recovery from fenced or prose-embedded contract JSON;
- scanning past unrelated and unmatched brace snippets;
- one structured-output retry before failure;
- opt-in worker-based JSON repair, recorded as degraded;
- a `claude-failover` replacement profile;
- fail-closed cross-process Claude invocation serialization.

Unix/macOS use `flock`; Windows uses a PID-file fallback. Serialization is
scoped to a state root, so processes using different `--state-root` values can
still overlap. It cannot detect Claude processes started outside Tagteam. The
Windows path is compile-checked but lacks equivalent runtime CI coverage.

## Installation and release integrity

The release supported normal `go install` and archive installation. Current
builds add embedded version, full commit SHA, build time, and dirty-tree state,
plus an adjacent checksum manifest.

Commands that can mutate a checkout reject missing/tampered manifests, dirty
release metadata, and ambiguous `dev` builds. Local development requires an
explicit `--allow-dev-build`. Read-only diagnosis remains available so a bad
installation can still be inspected with `version`, `verify-install`, `doctor`,
status, plan, transcript, and TUI views.

Release automation now builds and verifies manifests/checksums and applies
tighter workflow/release configuration than the first release.

## Internal architecture

The apparent source expansion is partly new behavior and partly decomposition.
At `v0.1.0`, `runner.go`, `runner_test.go`, `config.go`, and `config_test.go`
contained most behavior in very large files. Current `main` splits runner,
configuration, adapter, type, and test logic into bounded files and adds
dedicated modules for state, integrity, recovery, progress, findings, quality
gates, transfer, timeout calibration, durable I/O, and TUI behavior.

A CI line-count gate prevents Go files from exceeding 800 lines. This does not
change runtime semantics, but it makes orchestration and failure paths more
reviewable and reduces the risk of unrelated changes landing in one monolith.

manual playtests remain necessary for vendor CLI upgrades.

## Upgrade impact

Operators moving from `v0.1.0` should account for the following:

1. Do not assume authoritative artifacts are inside the checkout. Use the
   repository pointer or CLI inspection commands.
2. Review role defaults before running. Supervisor and relay defaults changed
   providers/models and relay retrieval is now off by default.
3. Expect stricter worker/reviewer output validation. Prose that previously
   passed through may now become an explicit contract failure.
4. Verified mutating commands require a valid release manifest or
   `--allow-dev-build` for a deliberate local development build.
5. Existing repo config remains readable, but high-authority keys from an
   untrusted `.tagteam.toml` may be ignored.
6. Use `resume` for interrupted authoritative runs rather than manually invoking
   the next role.
7. Use `findings` before transfer or closure; the latest run alone is not the
   complete unresolved-finding history.
8. Use separate Git worktrees for concurrent runs. Claude serialization does
   not make concurrent writers safe in one worktree.
9. Keep Claude-backed processes on one shared state root when serialization is
   required.

## Remaining limitations

Current `main` is substantially safer and more observable than `v0.1.0`, but
important limitations remain:

- vendor CLI flags, model identifiers, authentication, and output behavior can
  change independently of Tagteam;
- supervisor planning is more format-sensitive than final schema-validated
  review;
- Windows is compile-checked but not validated to the same runtime depth as
  macOS/Linux;
- the TUI has strong state/render unit coverage but no automated real-terminal,
  real-vendor end-to-end harness;
- openai-compatible remains a first-cut read-only reviewer/scout adapter;
- state-root-scoped Claude locking is not machine-global;
- real vendor timeout and contention behavior still requires playtesting.

## Bottom line

`v0.1.0` proved the multi-agent loop. Current `main` turns that loop into a
recoverable, inspectable, policy-gated execution system with a usable terminal
interface. The core modes are recognizable, but operational semantics are much
stricter: artifacts are external and authoritative, role output is validated,
partial work is recovered or quarantined, historical findings remain visible,
and applying work to another checkout requires explicit host verification.
